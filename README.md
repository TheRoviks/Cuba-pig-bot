# Кабанчик Куба

`Кабанчик Куба` это Telegram-бот для групповых чатов, который собирает цитаты из реплаев, хранит их в SQLite и при этом не смешивает данные между чатами. У каждого чата свой отдельный цитатник, а в личке с ботом пользователь видит только те данные, к которым у него реально есть доступ.

Бот уже умеет не только сохранять и показывать цитаты, но и вести нормальную служебную жизнь: обновлять реестр пользователей и чатов, логировать использование команд, писать события по цитатам и хранить аудит действий глобальных админов.

Подробно про запуск и команды: [COMMANDS.md](COMMANDS.md)

## Что умеет бот

- сохранять цитаты из реплая командой `/save_quote`
- показывать случайные цитаты через `/q`, `/quote`, `/quote_random`
- искать цитаты по тексту и по id
- показывать последние цитаты и локальный топ по чату
- удалять цитаты через soft delete, а не физически из базы
- работать сразу в нескольких чатах без протечки данных между ними
- показывать пользователю в личке только его доступные чаты и только его цитаты
- давать глобальным админам отдельную приватную админку
- собирать usage logs, quote events и admin audit logs

## Как устроена изоляция по чатам

В проекте одна база, но не один общий котёл.

Каждая цитата хранится с `chat_id`, а все обычные выборки по цитатам идут только внутри этого `chat_id`. Это касается:

- случайной цитаты
- поиска
- просмотра по id
- удаления
- локальных топов
- последних цитат

Если кто-то знает `quote_id` из другого чата, этого недостаточно: бот всё равно проверяет `chat_id` и чужую цитату не отдаёт.

## Личка и приватный кабинет

В личке бот работает как маленький кабинет пользователя.

Там можно:

- посмотреть общие чаты с ботом
- посмотреть сводку своих цитат по чатам
- открыть свои цитаты из конкретного чата

Доступ в личке строится не по догадкам, а через `user_chat_links`. Это значит, что пользователь видит только те чаты, где он реально был замечен, и только свои цитаты в этих чатах.

## Глобальная админка

У бота есть отдельная глобальная админка. Это не Telegram-admin в группе, а специальный список `user_id`, который задаётся в `.env` через `ADMIN_USER_IDS`.

Админка:

- работает только в личке с ботом
- недоступна обычным пользователям
- имеет отдельный аудит действий

Глобальный админ может:

- открыть панель
- посмотреть общую статистику
- посмотреть все чаты, где бот был замечен
- открыть статистику по конкретному чату
- посмотреть глобальные топы пользователей
- открыть health-статус базы и бота

## База данных

В основе проекта обычный SQLite, но схема уже не игрушечная.

### Канонические таблицы

- `users` — реестр пользователей, которых бот видел
- `chats` — реестр чатов
- `user_chat_links` — связь пользователь ↔ чат
- `quotes` — основное хранилище цитат

### Служебные и аналитические таблицы

- `quote_events` — события жизни цитаты: создание, показ, удаление, поиск
- `usage_logs` — использование команд
- `admin_audit_logs` — действия глобальных админов
- `app_meta` — служебные метаданные вроде версии схемы

### Что важно в схеме

- у цитат есть `UNIQUE(chat_id, original_message_id)` для защиты от повторного сохранения одного и того же сообщения
- удаление сделано через soft delete: `is_deleted`, `deleted_at`, `deleted_by_user_id`
- обычные запросы не показывают удалённые цитаты
- статистика и аудит живут отдельно, а не намешаны в основной таблице
- версия схемы хранится в `app_meta`

## Дубликаты и удаление

Дубликаты обрабатываются осторожно:

- сначала бот проверяет, не сохранялось ли уже это же сообщение в этом же чате
- потом может сделать вторичную консервативную проверку по `chat_id + quoted_user_id + текст`

Удаление цитаты не вычищает строку из базы. Бот просто помечает её как удалённую и записывает, кто и когда это сделал.

## Логирование

У бота есть три отдельных слоя логирования внутри базы:

- `usage_logs` для команд
- `quote_events` для событий вокруг цитат
- `admin_audit_logs` для действий глобальных админов

Это полезно для отладки, статистики и понимания того, что реально происходит в проекте.

## Безопасность

В проекте уже есть несколько практичных ограничений:

- чатовая изоляция всех запросов к цитатам
- проверка доступа в приватке через `user_chat_links`
- проверка admin-команд только в личке
- проверка `user_id` против `ADMIN_USER_IDS`
- параметризованные SQL-запросы
- проверка callback перед выполнением действий
- soft delete вместо жёсткого удаления
- простой cooldown на тяжёлые команды

## Поведение `/q`

В этом проекте `/q` это короткий алиас для случайной цитаты. Он не сохраняет цитату. Сохранение всегда делается через `/save_quote` из реплая.

## Для чего этот проект подходит

Этот бот подходит, если нужен живой цитатник для компании друзей или небольшого сообщества, где:

- в одном боте несколько чатов
- хочется личный кабинет в личке
- нужна базовая админ-аналитика
- важна изоляция чатов
- не хочется тащить отдельный сервер БД ради простой задачи
